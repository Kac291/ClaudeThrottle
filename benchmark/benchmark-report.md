## 评分标准

**准确性（1-5）：**
- 5 = 完全正确，无需修改
- 4 = 基本正确，有小瑕疵
- 3 = 大体正确，有明显问题需修改
- 2 = 部分正确，主要逻辑有误
- 1 = 基本错误或遗漏关键内容

## 关键发现与结论

### 成本节省最显著的场景

- **Economy 模式 L2 任务**：Haiku 替代 Sonnet 完成 7 个任务，准确性仅损失 1 分（T4 4/5），节省 79%
- **L1 任务全场景**：Haiku 在所有模式下处理搜索/提取，成本极低（单次 $0.05-0.06），质量基本无损

### 质量出现差异的场景

| 场景 | 模式 | 问题 |
|------|------|------|
| T4 测试脚本 | B Economy | Haiku 4/5：边界 case 覆盖不足 |
| T6 调试 | B Economy | Haiku 4/5：找到表层 bug，遗漏根因（SESSION_ID 注入问题） |
| T6 调试 | D Full Power | Opus 3/5：只分析不落地，未写回文件 |
| T2 风险提取 | D Full Power | Haiku 4/5：应对措施列有空洞 |
| T8 重构 | A/C | Sonnet/Opus 4/5：有效但偏保守 |
| T6 调试 | **E v2** | Haiku 4/5：找到空白字符问题（合理），但创建了额外未要求的脚本 |

### 最优模式推荐

| 使用场景 | 推荐模式 | 理由 |
|---------|---------|------|
| 代码探索/阅读为主 | **Economy** | 大量 L1 任务，Haiku 够用，成本极低 |
| 日常混合开发 | **Economy** | L2 质量差异微小（-1~2 分），成本降 79% |
| 关键架构/生产 bug | **Balanced** | L3 走 Opus 质量有保障，但需接受 3x 成本 |
| 成本极度敏感 | **Economy** | 唯一实现成本节省的模式 |

### 意外发现

1. **Balanced/Full Power 比无插件贵**：Opus 单价（$75/M out）是 Sonnet 的 5x，3 个 L3 任务产生的成本超过 L1/L2 节省的总和。若 L3 任务占比高，插件反而增加成本。

2. **Economy 模式 Haiku 处理 L2 质量接近 Sonnet**：48/50 与 A 组持平，说明大多数日常代码任务 Haiku 足够胜任。

3. **Full Power 路由不稳定**：T6（L2）被升 Opus，T8（L3）反而降为 Sonnet，路由规则在边界任务上表现不一致，且 Opus 升级未带来质量提升（T6 反而 3/5）。

4. **C 组 Haiku×12 异常**：Balanced 模式 2 个 L1 任务触发了 12 次 Haiku 调用，说明存在子任务分解过于细碎的情况，需进一步排查。

5. **缓存命中率高（92-95%）**：ClaudeThrottle 的规则文件被反复加载受益于 prompt cache，降低了实际 input 成本。

6. **E v2 修复 SESSION_ID bug**：v2 改为从 stdin JSON 提取 session_id，usage.log 从 `20260415`（日期）变为真实 UUID（`b33ac808-...`）。pre-tool-use.sh 新增 `|ok|` 状态字段，log 格式从 4 字段升为 5 字段。

7. **E v2 Haiku 调用纪律**：L1×2 + L2×5 = 7 次 Haiku，无拆分（vs C 组 L1×2 触发 12 次）。调用次数与任务数 1:1。

8. **E v2 Opus 拦截有效**：boost=off 全程，3 个 L3 任务均由 Sonnet 自执行，无 Opus 调用记录。拦截机制在 pre-tool-use.sh 层面正常工作。

9. **E v2 质量最高（49/50）**：v2 rules 更精确（单一策略），Haiku 调用纪律改善，L3 Sonnet 质量稳定，总分超过 B/C/D 组。
